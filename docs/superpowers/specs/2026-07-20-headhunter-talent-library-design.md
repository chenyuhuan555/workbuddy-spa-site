# Headhunter 人才库增强设计

## 状态

已确认，待实施。

## 目标

将简历上传、分类、岗位匹配与目标公司挖掘集中到 Headhunter 猎头工作台的人才库。保留单一人才主档和现有公司、岗位、推进记录关系，不复制 resume-matrix 的独立数据仓库。

## 范围

### 品牌

登录首页及可见品牌文案中的 `WorkBuddy` 改为 `Headhunter`。内部函数、存储键和兼容标识不改名，避免破坏既有数据与认证流程。

### 统一上传入口

人才库仅保留一个“上传简历”入口；移除“新建”菜单中的“上传人才简历”和“批量上传简历”。

同一弹窗包含：

- 选择或拖入一份/多份文件，复用现有批量队列、并发解析、逐项状态和查重决策。
- 直接录入区域，保留姓名、电话、邮箱、公司、职位、城市与粘贴简历文本。
- 单份和多份不再由用户选择不同入口；单份文件仍可使用人工字段补充与文本粘贴。

### 分类

- 分类目录保存在 `workbenchV2.settings.talentCategories`，结构为有序主分类与子分类。
- 人才主档保存可选的 `categoryIds`；一个人才可属于多个分类。
- 人才库支持按全部、主分类或子分类筛选；选中主分类时包含其子分类。
- 人才详情和直接录入区均可维护分类。
- 删除分类只移除关联的 `categoryIds`，不会删除人才、岗位或推进记录。
- 旧数据不迁移；现有候选人默认为未分类。resume-matrix 没有业务数据可导入。

### 岗位匹配

在人才库提供岗位匹配工作区：

- 可选择现有 Workbench 岗位，或临时录入/粘贴 JD。
- 可通过分类限定候选人范围。
- 使用现有 `WorkbenchV2.matchCandidates` / `matchPositions` 作为一致的基础评分，并补充技能命中、缺口与原因展示。
- 从现有岗位结果创建推进；临时 JD 只展示匹配结果，不自动创建公司、岗位或推进。

### 目标公司挖掘

- 以当前人才库筛选范围为输入，调用现有 DeepSeek 配置推荐目标公司。
- 排除候选人的当前/历史公司；显示行业、原因和关联人才。
- 支持根据选中公司生成 BD 方案。
- 结果为会话分析，不自动写入公司库；仅在用户显式选择创建时写入公司库。

## 约束与数据安全

- 不新增 IndexedDB store，不创建第二套人才库。
- 所有人才写入继续经 `WorkbenchV2.createTalent`、`updateTalent` 与 `appendTalentResumeVersion`。
- 岗位阶段仅保存在 `applications`，不得写入人才主档。
- 文件原件继续保存在已有 `files` store，快照只保存元数据。

## 验收

1. 人才库唯一入口可单份、多份、拖拽、粘贴文本与手动录入；逐项结果正确。
2. 新建菜单没有两项上传入口。
3. 分类可增删改排、可多选标记人才、筛选结果正确，删除分类不影响业务关系。
4. 现有岗位和临时 JD 都能返回可解释的匹配结果；仅现有岗位能创建推进。
5. 目标公司挖掘不自动建公司，显式创建才写入。
6. 登录首页显示 `Headhunter`，既有本地快照与认证路径仍可用。
